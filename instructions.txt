# YouTube Video Note Taker: SerpApi Migration and Refactoring Guide

## 1. Role and objective

You are working on an existing Node.js and Express project named `YoutubeVideoNoteTaker`.

The application currently accepts a YouTube URL, retrieves the video transcript, sends the transcript to an AI model, and returns generated study notes.

The current transcript implementation relies on direct YouTube access through experimental tools such as `yt-dlp`, audio downloads, caption scraping, or YouTube transcript libraries. These methods are unreliable on Render because YouTube blocks requests originating from shared datacenter IP ranges.

Your task is to replace the existing YouTube transcript acquisition system with the SerpApi YouTube Video Transcript API.

The finished application must:

1. Never download YouTube audio or video.
2. Never use browser cookies.
3. Never call YouTube directly from the server for transcript extraction.
4. Never expose the SerpApi API key to the browser.
5. Preserve the existing AI note-generation behavior.
6. Preserve the existing frontend unless a frontend change is required for correct error handling.
7. Remove or archive obsolete YouTube extraction experiments.
8. Include automated tests for the new transcript workflow.
9. Remain deployable on Render.
10. Use the SerpApi free allowance efficiently through caching, request deduplication, and input validation.

Do not rewrite the entire application unnecessarily. Inspect the existing implementation first and refactor incrementally.

# 5. SerpApi API behavior

Use the YouTube Video Transcript engine:

```text
engine=youtube_video_transcript
```

The required video identifier parameter is:

```text
v=<youtube-video-id>
```

Supported optional parameters relevant to this project include:

```text
language_code=en
type=asr
title=<specific-transcript-title>
no_cache=false
```

Do not set `no_cache=true` in the normal application flow.

SerpApi serves cached results only when all request parameters match. Its cache expires after one hour, and cached searches do not count against the monthly quota. Therefore, parameter values must be normalized and consistent. Do not randomly omit `language_code` on one request and include it on another for the same logical request.

The response can include:

```json
{
  "search_metadata": {
    "id": "...",
    "status": "Success"
  },
  "search_parameters": {
    "engine": "youtube_video_transcript",
    "v": "VIDEO_ID",
    "language_code": "en"
  },
  "transcript": [
    {
      "start_ms": 240,
      "end_ms": 7040,
      "snippet": "Transcript text",
      "start_time_text": "0:00"
    }
  ],
  "chapters": [
    {
      "chapter": "Introduction",
      "start_ms": 0,
      "end_ms": 214000
    }
  ],
  "available_transcripts": [
    {
      "language_name": "English",
      "language_code": "en",
      "type": "asr",
      "selected": true,
      "serpapi_link": "..."
    }
  ]
}
```

The official API returns timestamped transcript snippets, optional chapters, and available transcript tracks. When the requested language is unavailable, SerpApi documents that it may select the first available transcript language.

Do not assume that every successful HTTP response contains a non-empty transcript. Validate both the HTTP status and the JSON body.

# 9. SerpApi transcript service

Create:

```text
src/services/serpApiTranscriptService.js
```

The service must be the only part of the codebase that knows SerpApi request details.

Preferred public function:

```js
getYouTubeTranscript({
  videoId,
  languageCode,
  transcriptType
})
```

The controller must not construct SerpApi URLs.

## 9.1 Transport choice

First inspect the Node.js runtime.

When the project uses a Node.js version with stable built-in `fetch`, use native `fetch` and `AbortController` or `AbortSignal.timeout`.

Otherwise, use the official `serpapi` JavaScript package rather than an abandoned third-party wrapper. SerpApi provides an official JavaScript SDK with Promise and async/await support.

Avoid installing Axios solely for one request unless Axios is already part of the project.

## 9.2 Request construction

The server-side request should be equivalent to:

```js
const endpoint = new URL("https://serpapi.com/search.json");

endpoint.searchParams.set("engine", "youtube_video_transcript");
endpoint.searchParams.set("v", videoId);
endpoint.searchParams.set("language_code", languageCode);
endpoint.searchParams.set("api_key", config.serpApi.apiKey);
```

Do not include `type=asr` by default unless the user explicitly selected an auto-generated track or the existing product requires ASR transcripts specifically.

Do not include:

```text
no_cache=true
async=true
zero_trace=true
```

in the normal free-plan implementation.

Use deterministic parameters so repeat requests can benefit from SerpApi caching.

## 9.3 Request timeout

Every upstream request must have a timeout.

Default:

```text
20 seconds
```

The timeout must be configurable through `SERPAPI_TIMEOUT_MS`.

A timed-out request must not continue consuming server resources.

## 9.4 Retry policy

Retry only failures that may succeed on a second attempt:

```text
Network connection reset
Request timeout
HTTP 500
HTTP 503
Transient SerpApi processing error
```

Retry at most once by default.

Use a small exponential delay with jitter.

Do not retry:

```text
400 bad request
401 invalid API key
403 forbidden
404 invalid resource
429 quota or throughput exhausted
Transcript unavailable
Invalid YouTube input
```

Blindly retrying `429` wastes time and may worsen rate limiting.

## 9.5 Response validation

Validate:

```js
response.ok
body.search_metadata?.status
body.error
Array.isArray(body.transcript)
```

A usable result requires a non-empty transcript array containing at least one non-empty snippet.

Do not treat the following as success:

```json
{
  "search_metadata": {
    "status": "Success"
  },
  "transcript": []
}
```

SerpApi documents that a search can have `Success` status even when its result data is empty, so application-level validation remains necessary.

# 10. Normalized transcript model

Do not pass the raw SerpApi response throughout the application.

Normalize it immediately into an internal provider-independent object:

```js
{
  provider: "serpapi",
  videoId: "2QmuhAvJuLE",
  requestedLanguageCode: "en",
  resolvedLanguageCode: "en",
  transcriptType: "asr",
  transcriptTitle: null,
  segments: [
    {
      startMs: 240,
      endMs: 7040,
      startTimeText: "0:00",
      text: "Transcript text"
    }
  ],
  chapters: [
    {
      title: "Introduction",
      startMs: 0,
      endMs: 214000
    }
  ],
  availableTranscripts: [
    {
      title: null,
      languageName: "English",
      languageCode: "en",
      type: "asr",
      selected: true
    }
  ],
  fullText: "Transcript text",
  fetchedAt: "ISO_DATE",
  cacheHit: false
}
```

Normalization rules:

1. Convert all timestamps to finite numbers.
2. Rename `snippet` to `text`.
3. Trim each text segment.
4. Remove segments with empty text.
5. Preserve segment order.
6. Do not remove legitimate repeated phrases automatically.
7. Join `fullText` with a sensible separator.
8. Do not include `serpapi_link` in client responses.
9. Do not expose SerpApi search archive links.
10. Do not expose the API key.
11. Do not store the complete raw response unless debugging is explicitly enabled.
12. Do not log transcript contents in production.

Determine the resolved language from the selected entry in `available_transcripts` when available. Do not claim that the requested language was returned when SerpApi selected another language.

# 11. Transcript text preparation for AI notes

The AI note-generation service should receive normalized transcript data, not the raw SerpApi response.

Preferred input:

```js
generateStudyNotes({
  videoId,
  transcriptText,
  segments,
  chapters,
  languageCode
})
```

Preserve the current AI prompt unless it is clearly malformed.

When creating text for the AI:

1. Preserve paragraph or segment boundaries.
2. Optionally include timestamps when they improve note navigation.
3. Avoid JSON-stringifying the entire SerpApi response into the prompt.
4. Exclude search metadata and available track URLs.
5. Measure or estimate input size before calling the AI model.
6. Preserve existing chunking if the application already supports long videos.
7. Add chunking only when the current implementation can exceed the model context limit.
8. Do not truncate the end of the transcript silently.
9. If chunking is necessary, summarize chunks and then synthesize a final note set.
10. Keep the final note format compatible with the frontend.

A suitable transcript representation is:

```text
[00:00] First transcript segment.
[00:07] Second transcript segment.
[00:13] Third transcript segment.
```

When timestamps consume too many tokens, use plain joined text for summarization and retain timestamped segments separately for the UI.

# 12. Cache design

The free plan is limited, so caching is mandatory.

Implement a cache abstraction rather than embedding a raw `Map` inside the SerpApi service.

Suggested interface:

```js
getTranscriptCache(key)
setTranscriptCache(key, value, ttlMs)
deleteTranscriptCache(key)
```

Cache key:

```text
transcript:<videoId>:<normalizedLanguageCode>:<type-or-default>:<title-or-default>
```

Example:

```text
transcript:2QmuhAvJuLE:en:default:default
```

## 12.1 Initial implementation

A process-memory TTL cache is acceptable for the first implementation.

It must:

1. Expire entries.
2. Prevent unlimited growth.
3. Have a maximum number of entries.
4. Return copies or immutable data where practical.
5. Never include the API key.
6. Never cache errors permanently.

Recommended transcript cache TTL:

```text
24 hours
```

Make it configurable.

## 12.2 Render limitation

Do not use local files as the production cache.

Render instances may restart, redeploy, or run more than one process. A local JSON cache is not a reliable shared production cache.

Structure the cache module so it can later be replaced with Redis, Render Key Value, Upstash, or a database without changing controllers or the SerpApi service.

## 12.3 Negative caching

Short-term negative caching is allowed for clearly unavailable transcripts.

Example:

```text
TRANSCRIPT_UNAVAILABLE: cache for 5вЂ“15 minutes
```

Do not cache:

```text
Invalid API key
Quota exhausted
Network timeout
SerpApi server failure
Unexpected malformed response
```

for long periods.

# 13. Concurrent request deduplication

Caching alone does not prevent simultaneous duplicate requests.

If five users request the same uncached video at the same time, the application must send one SerpApi request, not five.

Implement an in-flight request map:

```js
const inFlight = new Map();
```

Use the same normalized cache key.

Algorithm:

```text
1. Check transcript cache.
2. If cached, return cached result.
3. Check whether the same key has an in-flight Promise.
4. If present, await the existing Promise.
5. Otherwise create the SerpApi Promise.
6. Store it in the in-flight map.
7. Cache the successful normalized result.
8. Remove the Promise from the map in a finally block.
```

Always remove failed and successful requests from the in-flight map.

Never leave a rejected Promise permanently stored.

# 16. Error taxonomy

Create a reusable `AppError` type with:

```js
{
  code,
  message,
  statusCode,
  retryable,
  cause
}
```

Map errors as follows.

## Invalid user input

```text
Code: INVALID_YOUTUBE_URL
HTTP: 400
Message: Enter a valid YouTube video URL.
Retryable: false
```

## Transcript not available

```text
Code: TRANSCRIPT_UNAVAILABLE
HTTP: 422
Message: No transcript is available for this video.
Retryable: false
```

The UI may suggest choosing another video or language.

## SerpApi key missing or invalid

```text
Code: TRANSCRIPT_SERVICE_CONFIGURATION_ERROR
HTTP: 503
Message: The transcript service is temporarily unavailable.
Retryable: false
```

Never tell the browser the API key is invalid.

Log the internal cause securely.

## SerpApi quota exhausted or hourly throughput reached

```text
Code: TRANSCRIPT_SERVICE_LIMIT_REACHED
HTTP: 503
Message: The transcript service has reached its usage limit. Please try again later.
Retryable: true
```

SerpApi uses HTTP `429` for both hourly throughput limits and exhausted search allowances.

## SerpApi timeout

```text
Code: TRANSCRIPT_SERVICE_TIMEOUT
HTTP: 504
Message: Transcript retrieval took too long. Please try again.
Retryable: true
```

## SerpApi server error

```text
Code: TRANSCRIPT_SERVICE_ERROR
HTTP: 502
Message: The transcript provider could not process this video.
Retryable: true
```

## AI generation failure

Keep this separate from transcript retrieval:

```text
Code: NOTE_GENERATION_FAILED
HTTP: 502
Message: The transcript was retrieved, but the notes could not be generated.
Retryable: true
```

This distinction is essential for debugging.

# 17. Rate limiting and free-quota protection

Add server-side rate limiting to the note-generation endpoint.

Make limits configurable through environment variables.

Initial default:

```text
10 requests per 15 minutes per IP
```

Adjust after observing real usage.

Rate limiting must occur after inexpensive structural checks where appropriate but before costly external calls.

Also implement:

1. Cache lookup before calling SerpApi.
2. Concurrent request deduplication.
3. Maximum request body size.
4. Maximum URL/input length.
5. Rejection of unsupported fields.
6. Optional daily application budget.
7. Logging of cache hit rates.
8. No automatic repeated transcript-language probing.

Do not perform multiple SerpApi requests merely to discover every language track.

Make one normalized request first. Only perform another request when the user explicitly chooses a different available transcript.

The free plan currently allows 250 searches per month and 50 requests of throughput per hour. Only successful searches count toward the monthly allowance; cached, errored, and failed searches are not counted according to the current pricing documentation.


```json
{

}
```

```text
api_key
account email
account ID
billing details
complete raw Account API response
```

Cache account-status checks for several minutes. Do not query them for every transcript request.

Do not make live SerpApi calls in the default test suite.

```text

```

Assert:

```text

Correct chapter mapping

```

Use a fixture where:


```json
    "status": "Success"
```

Expect:


Test a response containing:

```json

  "error": "..."
```

Assert that the internal error is classified and the raw message is not returned to the browser.

Mock HTTP `401`.

# 28. Acceptance criteria

* A valid captioned YouTube URL produces study notes.
* `youtu.be`, standard watch, Shorts, embed, and raw IDs are handled.
* Invalid URLs are rejected before SerpApi is called.
* The existing frontend remains usable.
* The existing AI provider still generates notes.
* Requested and resolved transcript languages are distinguished.
* Videos without transcripts produce a clear error.
* No audio or video is downloaded.

## Security

* The SerpApi key exists only in server-side environment configuration.
* `.env` is ignored.
* `.env.example` contains no real key.
* Browser responses contain no key.
* Logs contain no key.
* Logs do not contain complete transcript contents.
* Upstream error details are sanitized.

## Reliability

* SerpApi calls have a timeout.
* Transient failures retry no more than configured.
* `429`, `401`, and invalid input are not blindly retried.
* Empty transcript arrays are not accepted as success.
* Malformed responses do not crash the server.
* Centralized error middleware returns consistent JSON.
* The health endpoint does not depend on external APIs.

## Efficiency

* Successful transcripts are cached.
* Simultaneous duplicate requests are deduplicated.
* SerpApi caching remains enabled.
* Endpoint rate limiting is active.
* Live tests are disabled by default.
* No unnecessary language-probing requests occur.

## Cleanup

* Root `.mp3` and `.m4a` artifacts are gone.
* Obsolete extraction scripts are removed or deliberately archived.
* Unused YouTube downloading packages are removed.
* `package-lock.json` is synchronized.
